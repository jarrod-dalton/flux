# flux v1.10.2 is now live

This is a small **fluxCore-only patch release** that follows directly from
v1.10.1. It addresses an API ergonomics rough edge surfaced while polishing
tutorial 01: the only way to wire an in-memory `ModelBundle` into an `Engine`
was to wrap it in a do-nothing `ModelProvider` shim, which made every user's
*first* contact with `fluxCore` more confusing than it needed to be.

Only `fluxCore` and the super-repo move to `v1.10.2`. `fluxForecast` and the
other five subrepos remain at their v1.10.1 / v1.10.0 versions.

## What changed

### fluxCore (1.10.2)

- **New `Engine$new(bundle = ...)` shortcut.** The `Engine` constructor now
  accepts a `ModelBundle` directly, bypassing the `ModelProvider` machinery
  for inline / in-memory models:

  ```r
  # Before (v1.10.1 and earlier)
  toy_provider <- list(load = function(model_spec = NULL, ...) toy_bundle)
  eng <- Engine$new(provider = toy_provider)

  # After (v1.10.2)
  eng <- Engine$new(bundle = toy_bundle)
  ```

  When `bundle` is supplied, the engine validates it, stores it, and skips
  the `provider$load()` call entirely. `provider = ...` is fully unchanged
  and remains the right path for packaged or pluggable models (e.g.
  `PackageProvider$new(registry = list(my_model = build_my_model))`).
  Supplying both `bundle` and `provider` is an error.

- **`man/Engine.Rd`** gains a Constructor section documenting both
  construction paths and clarifies that `provider` and `model_spec` are
  `NULL` after a bundle-shortcut construction.

- **New tests** covering: bundle-shortcut runs end-to-end; bundle+provider
  conflict errors; invalid bundles rejected; legacy provider path unchanged.

### Tutorial 01 (`tutorials/01_core_engine_scaffold.Rmd`)

- Vignette renamed to "Engine and ModelBundle scaffold".
- Both `Engine$new(...)` call sites now use the new `bundle = ` shortcut.
- The `ModelProvider` discussion is now a single forward-pointer aside
  (production workflow: author from `fluxModelTemplate`, ship as an R
  package, register via `PackageProvider`); a dedicated providers tutorial
  is forthcoming.

### Super-repo

- `Imports: fluxCore (>= 1.10.2)`. All other dependency floors unchanged.

## Why this is a separate release rather than v1.10.1

The issue is real (the boilerplate is genuine cognitive cost on the user's
first encounter with `fluxCore`), but it is structurally distinct from
v1.10.1's scope (polish v1.10.0's headline schema features). v1.10.1 had
already been pushed when this was identified. v1.10.2 isolates the API
addition with its own NEWS entry, tests, and announcement.
