# flux v2.0.0

**Released:** 2026-06-02

flux v2.0.0 is a substantial rethinking of what a simulation model's structure
should look like. The core event engine that flux has always had — entity
timelines, probabilistic proposals, competing risks, irregular time — stays
exactly the same. What's new is everything that sits around it: how decisions
get modeled, how model parameters are organized, and how you assemble a model
in the first place.

---

## Decisions are now something a model declares

In v1.x, if your simulation needed to represent a clinical intervention or a
business decision — something like "when a patient reaches this disease state,
should we change their treatment?" — you either encoded that logic directly in
a transition function or wrote scaffolding around the engine yourself. There was
no standard way to do it.

In v2.0.0, **decision points** are declared on the schema alongside events and
state variables. A decision point says: "when this kind of event fires, ask the
policy whether to take an action." The policy is its own function — it looks at
the entity's current state and returns a choice. The engine handles everything
else: injecting the action into the event timeline, firing the right handler,
and building a record of what happened.

The record part matters. Every decision point activation writes a
**trajectory record** — what state the entity was in, what the policy observed,
what it proposed, what was actually realized. At the end of a run, you can pull
those records out as a tidy table. That's your decision audit trail, and it
comes for free.

For a full walkthrough, including a complete example of a policy-driven delivery
simulation with two competing route options, see [Tutorial 03: Decisions and
Policy](../../tutorials/03_decisions_policy.md#decision-points-and-policies).

## Cleaner model parameters

In v1.x, when you wanted parameter uncertainty — different coefficients per draw
in a Bayesian analysis, for example — you had to build the crossing yourself.
The bundle would receive a generic `ctx` argument and hope that whatever was
passed in had the right shape.

v2.0.0 makes this explicit. Your bundle can declare a **`sample_params`** hook:
a function that draws `n` parameter sets and returns them as a list of
**`ParamContext`** objects. Each `ParamContext` holds one draw's worth of
parameters — a named list you define — plus a draw ID for tracking. When you
run a cohort with `n_param_draws`, the engine calls your hook, then runs every
entity under every parameter draw, fully crossing entities × draws × stochastic
replicates. No additional plumbing on your end.

The typed `ParamContext` object also replaces the old freeform `ctx` argument
for parameter-carrying callbacks. The engine will now give you an error at
construction time if a callback still has `ctx` in its signature — a clean
break rather than a silent behavior difference.

For a worked example showing `ParamContext` construction, the `sample_params`
hook, and how to inspect per-draw results, see the [parameter uncertainty
section of Tutorial 01](../../tutorials/01_core_engine_scaffold.md#incorporating-parameter-uncertainty).

## Assembling a model: `load_model()`

Previously, wiring together a schema, a model bundle, a policy, and runtime
configuration required calling `Engine$new()` directly and managing the order
of operations yourself. In v2.0.0, **`load_model()`** does this as a validated
assembly step. It takes your components, checks that they're mutually consistent
(schema and bundle callbacks are compatible, decision points referenced by the
policy actually exist, and so on), and returns a configured Engine. For simple
cases — no policy, no runtime config — `Engine$new(bundle = my_bundle)` still
works fine. But as models get more complex, `load_model()` gives you a reliable
single entry point.

## Other improvements

**`refresh_rules()`** lets a bundle declare which simulation processes should
re-propose after each event fires. By default, all processes re-evaluate after
any event — that's the right behavior for most models. But if you have a process
that should only fire once, or one that should only re-evaluate after specific
event types, `refresh_rules` lets you say that explicitly in one place, rather
than encoding it in individual transition or stop functions. See the [refresh
rules section of Tutorial 01](../../tutorials/01_core_engine_scaffold.md#using-refresh-rules).

**Derived variables** let a schema variable be computed from the entity's state
at read time, rather than stored. This is useful for quantities that can always
be derived from what's already tracked — a current age, a time-since-event, a
composite severity score. The schema declares the derivation function; entity
code reads the variable normally.

---

## Upgrading from v1.x

The engine API is backward compatible for bundles that don't use `ctx`. If your
bundle callbacks have `ctx` as a formal argument, the engine will error on
construction and tell you which callbacks to update. The fix is mechanical:
replace `ctx` with `param_ctx = NULL`, `sim_ctx = NULL`, or `runtime_ctx = NULL`
depending on what you were passing through it.

If you were using `ModelProvider`, `PackageProvider`, or `Engine$new(provider=)`,
those interfaces are removed. The replacement is `Engine$new(bundle = ...)` or
`load_model()`.
