## TL;DR

v2.0.0 adds a formalized action/policy layer, replaces `ctx` with explicit typed
context objects, standardizes reproducibility, and positions the ecosystem for
cross-language portability and future ABM/RL extension.

Primary target: **fluxCore**. Downstream packages (fluxForecast, fluxOrchestrate,
fluxPrepare, fluxValidation) updated after fluxCore contracts stabilize.

**Locked decisions (do not re-open without filing a counter-proposal on issue #1):**
- Engine = Schema + ModelBundle + optional runtime components
- `ModelProvider` removed from first-class architecture; replaced by `load_model()`
- `ctx` removed as first-class interface; formal typed contexts replace it
- No user-facing `ctx` backward compatibility in v2.0.0; fail fast on `ctx`-style usage
- Portability = portable data specs + language-native function implementations
- Backward compatibility is secondary in v2.0.0; proper back-compat begins in v2.x
- No bespoke RNG/seed formulas in public API; standardized `seed=` argument everywhere
- Decision points must be declared explicitly in Schema, not inferred at runtime

## Architecture Mental Model

```
load_model(
  schema,                   # required -- portable, data-only
  bundle,                   # required -- language-native functions
  policy        = NULL,     # optional -- action proposals at decision points
  environment   = NULL,     # optional -- external signals for ABM/RL
  trajectory    = NULL,     # optional -- canonical decision/action logging
  runtime       = NULL,     # optional -- seed, backend, workers (defaults applied)
  param_source  = NULL      # optional -- resolves ParamContext once per run
)
  returns Engine

Engine$run(entity)
  returns list(entity, observations, trajectory_records)

run_cohort(engine, entities, n_param_draws, n_sims, runtime, ...)
  returns cohort results
```

### Engine vs load_model()

`load_model()` is the **assembly and validation** function. It validates all
components against the schema and each other, applies defaults, and returns a
configured Engine.

`Engine` is the **runtime object** that executes simulations.

Both exist. `load_model()` is the recommended Level 1/2 entry point; advanced
users can construct Engine directly for maximum control.

`ModelProvider` is **removed** as a first-class concept. Package/file/registry
loading becomes an optional utility wrapper around `load_model()`, not an
architectural requirement.

## User Experience Tiers

| Level | What the user touches                           | Where model lives           |
|-------|-------------------------------------------------|-----------------------------|
| 1     | schema + bundle in-memory, load_model(), run()  | R or Python session         |
| 2     | schema/spec exported to JSON; bundle stays local| JSON files + language code  |

Level 1 must remain as frictionless as current v1.x. All advanced components opt-in

## Contract Tables (Stage 1 — signed off)

### Schema

| Field            | Type                   | Required | Notes                                        |
|------------------|------------------------|----------|----------------------------------------------|
| variables        | list of variable specs | yes      | types, defaults, allow_na (existing)         |
| event_catalog    | list of event specs    | yes      | event_type labels + metadata                 |
| time_spec        | time_spec object       | yes      | unit, origin, zone                           |
| terminal_events  | character vector       | no       | events that halt simulation                  |
| decision_points  | list of DecisionPoint  | no       | empty = no-policy model (Level 1/2 path)     |
| observation_spec | list                   | no       | schema for observe() output                  |
| param_schema     | list                   | no       | expected structure of parameter realizations |

Schema is the **portable, data-only** component. Round-trips cleanly to/from JSON.
It is the piece that can be shared across languages without function translation.

### ModelBundle

| Callback       | Signature                                    | Required | Notes                             |
|----------------|----------------------------------------------|----------|-----------------------------------|
| propose_events | (entity, sim_ctx, param_ctx)                 | yes      | returns list of proposals         |
| transition     | (entity, event, sim_ctx, param_ctx)          | yes      | returns state patch               |
| stop           | (entity, sim_ctx)                            | yes      | returns logical                   |
| observe        | (entity, event, sim_ctx, param_ctx)          | no       | model-defined; returns rows/NULL  |
| refresh_rules  | (entity, last_event, changes, sim_ctx)       | no       | returns "ALL" or process_ids      |
| init_entity    | (entity, sim_ctx, param_ctx)                 | no       | one-time setup hook               |

Bundle is **language-native** — R functions in R, Python callables in Python.
Functions receive typed context objects, not a catch-all `ctx`.

Key param principle: bundle functions do NOT call `param_source` directly.
`param_source` resolves once per run into a `ParamContext`, which is injected into
callbacks. The same separation principle applies to environment, policy, runtime.

### RuntimeContext

| Field        | Type      | Required | Notes                                             |
|--------------|-----------|----------|---------------------------------------------------|
| seed         | integer   | no       | top-level seed; NULL = unseeded                   |
| replicate_id | integer   | no       | stochastic replicate index within a draw          |
| backend      | character | no       | "none" (default), "cluster", "mclapply", "future" |
| n_workers    | integer   | no       | workers for parallel backends                     |
| stream_id    | integer   | no       | internal; set by run harness, not user            |

RNG contract (public): fixed seed + draw_id + replicate_id + entity_id = reproducible
output. Internal stream allocation is an implementation detail, not a public formula.

Reproducibility use cases:
- single run:      `seed=` in `Engine$run()` or RuntimeContext
- repeated batch:  `seed=` in `run_cohort()` or RuntimeContext
- counterfactual:  identical seed + draw_id + entity_id across scenario variants
- parallel:        RuntimeContext handles backend-safe stream splitting internally

### DecisionPoint (declared in Schema)

| Field           | Type                  | Required | Notes                                         |
|-----------------|-----------------------|----------|-----------------------------------------------|
| id              | character             | yes      | unique, e.g., "post_dropoff"                  |
| trigger         | character or function | yes      | event_type name(s) or predicate function      |
| allowed_actions | character vector      | no       | named action types; NULL = unconstrained      |
| observation_fn  | function              | no       | computes observable inputs; default: snapshot |
| label           | character             | no       | human-readable description                    |

Decision points are explicitly declared in the schema — not inferred at runtime.
This makes them inspectable, testable, portable, and safe for trajectory logging.

**Urban food delivery example** (declared in `schema$decision_points`):

```r
post_dropoff_dp <- list(
  id              = "post_dropoff",
  trigger         = "dropoff",
  allowed_actions = c("accept_order", "charge", "reposition", "idle"),
  observation_fn  = function(entity) list(
    status      = entity$get("status"),
    battery_pct = entity$get("battery_pct"),
    payload_kg  = entity$get("payload_kg"),
    route_zone  = entity$get("route_zone"),
    prob_rain   = entity$get("prob_rain")
  ),
  label = "Courier dispatch decision after completing a delivery"
)
```

At simulation time 8.4, after a `dropoff` event:
1. Engine detects decision point trigger
2. `observation_fn` called → `{status=at_depot, battery_pct=0.22, ...}`
3. Policy consulted → proposes `ActionEvent(action_type="charge", time_next=8.4)`
4. ActionEvent inserted into canonical timeline (same arbitration as all events)
5. TrajectoryRecord emitted (if logger configured)
6. Simulation continues

### D/P/A Responsibility Map

Decision/Policy/Action functionality is distributed across packages along three
orthogonal dimensions:

```
                  Single model          Multi-model
               ┌──────────────────┬──────────────────────┐
 Single entity │   fluxCore       │   fluxOrchestrate    │
               │   Engine + policy│   multi-model routing │
               ├──────────────────┼──────────────────────┤
 Multi-entity  │   fluxSimulate   │   fluxSimulate +     │
               │   + fluxCore     │   fluxOrchestrate    │
               └──────────────────┴──────────────────────┘

               ×  Single policy: run one policy through Engine
               ×  Alternative policies: fluxSimulate (policy comparison,
                  counterfactual rollouts, RL training loops)
```

**Package responsibilities:**

| Package | Owns | Does NOT own |
|---|---|---|
| **fluxCore** | DecisionPoint declaration in Schema; ActionEvent as timeline event; TrajectoryRecord emission; single-entity single-policy dispatch via `load_model(policy=...)` | Multi-entity coordination; policy comparison; RL training |
| **fluxOrchestrate** | Multi-model event routing for one entity (eligible_models, event_priority, on_transition, stop); merged-schema arbitration | Treatment policy dispatch; multi-entity coordination |
| **fluxPrepare** | Decision-anchored TTV datasets (`build_ttv_decision`); action-conditioned feature engineering surfaces | Policy evaluation; simulation |
| **fluxValidation** | Retrospective observed-vs-predicted comparison (future: observed-vs-policy action alignment) | Policy optimization |
| **fluxForecast** | Risk/survival curves under a given policy; dual mode (strict/exploratory) when decision_points exist | Policy search; counterfactual comparison |
| **fluxSimulate** *(future sub-repo)* | Multi-entity environment management; time-axis alignment across entity types; cross-entity decision points; policy comparison / counterfactual rollouts; RL episode lifecycle; reward aggregation | Model definition; data preparation; validation |

**fluxSimulate** (working name) is the planned add-on sub-repo for capabilities
that span multiple entities, multiple policies, or both. It imports fluxCore
(Engine, Entity, DecisionPoint) as foundations and optionally imports
fluxOrchestrate when multi-model entities participate. The name reflects that
this package owns the simulation campaign layer — running entities through
environments under alternative policies — rather than any single narrower
concern (environment management, arena, RL).

fluxSimulate scope includes:
- **Multi-entity ABM**: time-axis alignment, cross-entity decision point surfacing,
  shared environment state (EnvironmentContext dispatch)
- **Policy evaluation**: run the same cohort under alternative policies; compare
  TrajectoryRecords across scenarios
- **Counterfactual rollouts**: fix seed + entity + draw, vary policy; diff outcomes
- **RL training loops**: episode management, reward aggregation, policy gradient
  interfaces (algorithm implementations are out of scope for v2.0.0)

fluxSimulate is **not required for v2.0.0 release**. The core D/P/A loop
(Schema → Engine → policy → ActionEvent → TrajectoryRecord) is fully functional
in fluxCore today. fluxSimulate extends this to multi-entity and multi-policy
scenarios.

### ActionEvent (timeline-native)

| Field             | Type       | Required | Notes                                     |
|-------------------|------------|----------|-------------------------------------------|
| action_type       | character  | yes      | must match allowed_action if declared     |
| time_next         | numeric    | yes      | realization time on canonical timeline    |
| decision_point_id | character  | yes      | which decision point produced this        |
| params            | named list | no       | action-specific parameters                |
| metadata          | named list | no       | policy provenance or audit fields         |

ActionEvents are timeline-native: they enter the same arbitration and refresh
lifecycle as all other events. No side-channel state mutation.

### TrajectoryRecord (engine-owned)

| Field             | Type             | Required | Notes                                  |
|-------------------|------------------|----------|----------------------------------------|
| run_id            | character        | yes      |                                        |
| entity_id         | character        | yes      |                                        |
| t                 | numeric          | yes      | time of decision point                 |
| decision_point_id | character        | yes      |                                        |
| observation       | named list       | yes      | output of observation_fn               |
| candidate_actions | character vector | no       | presented to policy                    |
| proposed_actions  | list             | no       | what policy proposed                   |
| selected_action   | ActionEvent      | no       | what was realized on timeline          |
| realized_event    | event record     | yes      | event that triggered decision point    |
| state_before      | named list       | no       | NULL default; configurable             |
| state_after       | named list       | no       | NULL default; configurable             |
| reward            | numeric/list     | no       | optional reward signal(s)              |

Relationship to existing surfaces:
- `snapshot()` / `snapshot_at()`: ad hoc state reconstruction — unchanged
- `observe()`: model-defined streaming output — unchanged, arbitrary structure
- TrajectoryRecord: engine-owned, standardized, emitted only at declared decision points

TrajectoryRecords are the canonical surface for policy evaluation, audit, and RL.
`state_before`/`state_after` are NULL by default; trajectory logger config controls detail.

---

### ParamSource

| Method         | Signature                     | Notes                                    |
|----------------|-------------------------------|------------------------------------------|
| sample(n)      | (n) -> list\<ParamContext\>   | n independent parameter realizations    |
| fixed(params)  | (list) -> ParamContext        | wrap a fixed set as draw_id=1            |
| from_posterior | (matrix) -> list\<ParamContext\> | wrap posterior sample rows            |

Default: if `param_source` is NULL and `param_schema` defined, bundle defaults used.
If neither defined, `params = empty list`.

---

### load_model() Validation Rules

1. All variable types referenced in bundle outputs must exist in `schema$variables`
2. All event types proposed by bundle must exist in `schema$event_catalog`
3. All `action_type` values from policy must exist in declared `allowed_actions`
4. If `trajectory` is non-NULL, `schema$decision_points` must be non-empty
5. `runtime` seed + backend combination validated for backend safety
6. `param_source` output shape must conform to `schema$param_schema` if both defined
